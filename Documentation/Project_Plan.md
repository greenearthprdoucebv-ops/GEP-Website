## 1. Project Overview

This project plan focuses on the first part of our PRJ4 project: the development of the website.  
The website will serve as the main digital platform for the project and should present the company, its products/services, and relevant user-oriented information in a clear and professional way.

---

## 2. Project Goal

The goal of the website phase is to create a functional and user-friendly website that:

- represents the project/company in a professional way
- communicates important information clearly
- supports the needs of the target group
- provides a solid basis for further project development
- can be improved iteratively based on feedback

---

## 3. Scope

### In Scope
- requirements gathering for the website
- identifying target users and their needs
- content planning and page structure
- low-fidelity and/or high-fidelity design
- frontend development of the website
- responsive layout for desktop and mobile
- basic navigation and interaction
- feedback processing and iteration
- testing and improvement

### Out of Scope
- advanced backend integration
- complex database functionality
- login systems, unless required later
- non-website project parts such as other platforms or extensions

---

## 4. Relation to LO1

In this project phase, we focus on LO1 by planning and executing the website development in an agile way.  
Our work is divided into clear activities on specific layers:

- **Analysis** – identifying stakeholders, goals, requirements, and website content
- **Design** – creating structure, wireframes, visual ideas, and user flow
- **Implementation** – building the website frontend and improving it through feedback

This approach helps us show not only the final product, but also our planning, progress, decisions, and iterations, which are important for the portfolio and evidence collection.

---

## 5. Main Activities

### Activity 1 – Analysis
In this activity, we determine what the website should achieve and what users need from it.

**Tasks:**
- identify the purpose of the website
- define the target audience
- gather stakeholder wishes and requirements
- analyze similar websites for inspiration
- determine required pages and content
- write down functional and non-functional requirements

**Deliverables:**
- requirement list
- target group description
- competitor/inspiration analysis
- sitemap or page overview

---

### Activity 2 – Design
In this activity, we transform the requirements into a concrete website concept.

**Tasks:**
- create page structure and navigation
- make sketches or wireframes
- decide on layout, colors, typography, and style
- define reusable UI elements
- review the design with the team and improve it

**Deliverables:**
- sitemap
- wireframes/mockups
- style direction
- design decisions overview

---

### Activity 3 – Implementation
In this activity, we build the website and improve it step by step.

**Tasks:**
- set up the project structure
- create the main pages
- implement navigation
- build responsive layouts
- add content and visuals
- test the website on different screen sizes
- fix issues and refine based on feedback

**Deliverables:**
- frontend code repository
- working website prototype
- test notes
- updated version after feedback

---

## 6. Agile Working Method

We will work in an agile way by dividing the website development into smaller steps.  
Each week, we define what needs to be done, implement a part of the website, review progress, and improve based on feedback.

### Our agile approach includes:
- short planning per week
- task division within the team
- regular check-ins
- iterative improvements
- using feedback from coach, stakeholders, and team members
- documenting decisions and evidence for the portfolio

## 9. Roles and Task Distribution

Each team member contributes to the website phase in a specific way.  
The exact distribution can be adjusted depending on team size and strengths.

**Possible task distribution:**
- team member 1: requirements and documentation
- team member 2: wireframes and design
- team member 3: frontend implementation

Even when tasks are divided, all team members stay involved in review and decision-making.

---

## 10. Risks and Solutions

| Risk | Possible Impact | Planned Solution |
|------|-----------------|------------------|
| unclear requirements | wrong implementation direction | discuss early and document requirements clearly |
| too many design changes | delays in development | freeze core structure before implementation |
| uneven task division | overload for some team members | divide tasks clearly and review progress weekly |
| technical frontend issues | slower implementation | start with a simple structure and improve step by step |
| lack of feedback integration | weaker final quality | collect feedback regularly and document improvements |

---

## 11. Quality Criteria

The website part will be considered successful if it:

- has a clear structure and navigation
- is visually consistent
- is responsive on common screen sizes
- reflects the project/company professionally
- matches the main requirements
- shows visible improvement after feedback
- is documented well enough to serve as portfolio evidence

---

## 12. Evidence for Portfolio

During this project phase, we will collect evidence that shows our planning, work process, and progress.

**Examples of evidence:**
- requirement documents
- meeting notes
- sitemap
- wireframes/mockups
- GitHub commits
- screenshots of development progress
- feedback notes
- improved versions after feedback
- testing notes
- reflection on what was learned and improved

Strong evidence should not only show the artifact itself, but also explain what was done, why it matters, and how it supports the learning outcome.
